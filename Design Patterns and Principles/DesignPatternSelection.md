# When to Use Design Patterns

## Singleton Pattern

Use when exactly one instance of a class should exist and be shared across the entire application.

## Factory Method Pattern

Use when the type of object to create is not known beforehand or may vary based on conditions.

## Builder Pattern

Use when creating complex objects with many optional properties or configuration steps.

## Adapter Pattern

Use when two classes or systems have incompatible interfaces but need to work together.

## Decorator Pattern

Use when additional behavior or functionality needs to be added to objects dynamically without changing their original code.

## Proxy Pattern

Use when access to an object needs to be controlled, restricted, delayed, or monitored.

## Observer Pattern

Use when multiple objects need to be automatically informed whenever another object's state changes.

## Strategy Pattern

Use when multiple algorithms or behaviors can perform the same task and the behavior may change at runtime.

## Command Pattern

Use when requests or actions should be encapsulated as objects, allowing execution, queuing, logging, or undo operations.

## Model-View-Controller (MVC)

Use when you want to separate application data, business logic, and user interface into independent components.

## Dependency Injection

Use when classes depend on external services or components and you want to reduce tight coupling between them.
